# 008 – 21/04/2026 – Case Hell: unify on camelCase app-side

## Problem

The project mixes three conventions and that mix is starting to hurt:

1. **Database**: `snake_case` everywhere (`user_id`, `first_name`, `created_at`, …). Enforced by migrations — this is the source of truth and will not change.
2. **Internal TypeScript code**: currently `snake_case` too, to match DB rows (see `src/types/*.ts`, route handlers, schemas, seed helpers).
3. **HTTP boundary**: `camelCase`, enforced by `middleware/caseConverter.ts` (`camelToSnake` on request bodies, `snakeToCamel` on response bodies).

Consequences:

- Two conventions collide inside the same file — e.g. `PersonaConfiguration` uses `first_name` and `last_name` but also `clubCode`, `categoryName`, `assignments`. Authors don't know which to pick.
- Zod schemas for payloads use snake_case internally but external callers see camelCase (the middleware flips them) — the schema never matches what hits the wire.
- JSX/React consumers of API responses are camelCase, but anyone writing a route has to mentally translate to snake_case and back.

## Target state

**One convention app-side: `camelCase` everywhere except inside migrations and raw SQL.**

- **Database**: unchanged. Still snake_case.
- **Internal TypeScript code**: 100% `camelCase` — row types, route handlers, Zod schemas, seed helpers, config objects.
- **DB ↔ app boundary**: handled by Knex hooks, not by hand.
- **HTTP boundary**: no conversion needed. Requests come in camelCase, internal code uses camelCase, responses go out camelCase.
- **`middleware/caseConverter.ts`**: deleted.

## Approach

Use Knex's built-in `wrapIdentifier` + `postProcessResponse` hooks in `apps/api/src/db.ts`:

```ts
import knex from 'knex';
import camelcaseKeys from 'camelcase-keys';
import { snakeCase } from 'change-case';

const db = knex({
  client: 'pg',
  connection: process.env.DATABASE_URL,
  wrapIdentifier: (value, origImpl) => origImpl(snakeCase(value)),
  postProcessResponse: (result) =>
    Array.isArray(result)
      ? result.map((row) => camelcaseKeys(row, { deep: true }))
      : camelcaseKeys(result as Record<string, unknown>, { deep: true }),
});
```

**Why hooks and not a mapper layer?**

- Zero per-repository boilerplate — no `mapUserRow`, no `mapMemberRow`.
- Applies uniformly to every query, including ad-hoc ones in scripts and tests.
- Knex's query builder identifiers (`where`, `insert`, `update`, `orderBy`, …) go through `wrapIdentifier`, so `db('users').where({ firstName: 'x' })` becomes `... WHERE first_name = 'x'` transparently.

**Why migrations are safe.**

Migrations run via the `knex` CLI using `knexfile.js`, which is a **separate Knex instance** with no hooks. So `t.text('first_name')` in a migration stays `first_name`. No migration needs to change.

## Libraries

- `camelcase-keys` (Sindre Sorhus) — deep object-key transform, returns new objects. Used in `postProcessResponse`.
- `change-case` (or just the `snake-case` export from it) — used in `wrapIdentifier`. The existing `caseConverter.ts` hand-rolls a regex, but a vetted lib handles edge cases like acronyms, numbers, already-snake input.

Both are dependency-light and widely used.

## Step-by-step

1. **Wire the hooks in `src/db.ts`** and add `camelcase-keys` + `change-case` to `apps/api/package.json`. Smoke-test with one route to confirm the round-trip works.
2. **Flip row types in `src/types/*.ts`** to camelCase — `User`, `Member`, `Club`, `Team`, `TeamCategory`, `TeamAssignment`, `MemberStatus`, `UserMember`. Fields like `user_id` → `userId`, `created_at` → `createdAt`, etc.
3. **Update route handlers** (`src/routes/**/*.ts`) — `.select()`, `.where()`, `.insert()`, `.update()` arg keys, plus all `.rows[i].some_field` accesses. Target dirs: `auth`, `clubs`, `invitations`, `member-statuses`, `members`, `teams`, `user-members`, `users`.
4. **Update Zod schemas** (`src/schemas/*.ts`) — field names flip to camelCase. Because the middleware is going away, the schema now describes exactly what hits the wire.
5. **Delete `middleware/caseConverter.ts`** and remove the two `app.use(...)` wires in `src/app.ts`.
6. **Update seed scripts** (`src/scripts/**/*.ts`) — `seedCreate`, `seedUtils`, `seedConfiguration`, `personasConfiguration`. Fix the mixed objects along the way.
7. **Update tests** (`src/routes/**/*.test.ts`, `test-utils.ts`) — request/response bodies and DB assertions.
8. **Audit `knex.raw` / `db.raw` calls** — only two in app code (`seedUtils.ts:20` `SELECT 1`, `routes/health.ts:29` `SELECT NOW()`), both safe. Migrations use raw SQL extensively but run through `knexfile.js` → unaffected.
9. **Verify `pnpm lint` + `pnpm --filter @eggplant/api exec tsc --noEmit` + `pnpm --filter @eggplant/api test` all pass.**
10. **Manual smoke test**: `make migrate`, `pnpm --filter @eggplant/api seed-create --force`, hit a couple of routes via Bruno to verify request/response shapes still look like camelCase externally.

## Blast radius

From the survey: **261 snake_case identifier hits across 50 files** (routes, schemas, scripts, types, tests). Migrations untouched.

## Risks and mitigations

- **`wrapIdentifier` mangles SQL fragments.** Only matters for raw SQL — we verified the two app-level `raw()` calls are safe. JSON path expressions and window functions aren't used.
- **Already-snake_case identifiers passed to the query builder** (e.g. copy-pasted `.where({ user_id: ... })` left over) — `snakeCase('user_id')` returns `user_id` unchanged, so this is a warning to grep for during the refactor but not a correctness issue.
- **Response caching or third parties depending on snake_case fields** — none in this codebase; the HTTP boundary has always been camelCase thanks to the middleware.
- **Scripts that write to DB outside Knex** — none; `pg` is only used transitively via Knex.

## Commit strategy

Split along step boundaries to keep diffs reviewable:

- `refactor(db): add camelCase↔snake_case Knex mappers`
- `refactor(types): flip row types to camelCase`
- `refactor(routes): use camelCase field names`
- `refactor(schemas): align Zod schemas with camelCase wire format`
- `refactor(http): remove caseConverter middleware`
- `refactor(scripts): flip seed scripts to camelCase`
- `test: update fixtures to camelCase`

## Out of scope

- PWA app — already camelCase, no change.
- Renaming DB columns — explicitly not happening. `snake_case` stays in the DB.
- Changing the OpenAPI annotations or Bruno fixtures, *unless* a route's wire shape was accidentally wrong before (the middleware already made requests/responses camelCase, so the wire shape isn't changing).
